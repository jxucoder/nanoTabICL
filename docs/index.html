Stage 1

Understand each feature.

Let's trace one cell through the full column-embedding pipeline step by step, using a tiny concrete example.

0

Our example

A tiny table: 3 rows, 2 features, binary labels. The first two rows are training (labeled), the last is a test row (unlabeled). We'll follow the highlighted cell 0.3 through every operation.

x1x2y
train0.3−1.20
train−0.70.41
test0.10.8?
1

Isolate each column

The table is transposed so each feature column becomes its own independent group. Columns never see each other during Stage 1 — they share the same encoder weights but are processed separately.

Column x1

0.3 −0.7 0.1

Column x2

−1.2 0.4 0.8

From here on we follow column x1 only. Column x2 goes through the exact same steps in parallel.

2

Project each scalar to 64 dimensions

A learned linear layer Linear(1, 64) maps every single number to a 64-dimensional vector. This is the same layer for every cell in every column.

0.3 → Linear(1, 64) → [ 0.12, −0.41, 0.08, … ]64 numbers
−0.7 → Linear(1, 64) → [−0.29, 0.96, −0.18, … ]64
0.1 → Linear(1, 64) → [ 0.04, −0.14, 0.03, … ]64

Same weights, different inputs → different vectors. At this point the vectors carry no label information.

3

Add label encoding (train rows only)

The label is one-hot encoded, then projected to 64 dims through a separate learned layer Linear(C, 64). This label vector is added element-wise to the value projection. Test rows skip this — no label to add.

[ 0.12, −0.41, … ] +
y=0 [1,0,…] → Linear(C,64) [ 0.05, 0.33, … ]
= [ 0.17, −0.08, … ]
[−0.29, 0.96, … ] +
y=1 [0,1,…] → Linear(C,64) [−0.22, 0.14, … ]
= [−0.51, 1.10, … ]
[ 0.04, −0.14, … ] no label → unchanged [ 0.04, −0.14, … ]

The same label vector is added to every feature column for that row, so x2 gets the same y=0 / y=1 encoding added.

4

Let every value see the rest of the column (ISAB)

Right now each of our 3 vectors was built in isolation — the vector for 0.3 knows nothing about −0.7 or 0.1. We want each vector to be aware of the other values in the same column, so the model understands the feature's distribution (range, spread, outliers).

"Attend to" means: look at another vector, decide how relevant it is (a weight between 0 and 1), then mix a weighted combination of it into yourself. It's the same Query-Key-Value mechanism used in transformers and LLMs — applied here to the values within a single column.

Concrete example — what happens to the vector for 0.3

If we did plain self-attention, 0.3's vector would look at the vectors for −0.7 and 0.1, compute a similarity score with each (via dot product), turn those scores into weights (via softmax), and add a weighted mix of their values into itself. After that, 0.3's vector "knows" it sits in a column whose values range from −0.7 to 0.3, that the spread is moderate, etc.

Why not just do that directly?

We could, but with many rows it's expensive — every value would look at every other value (n² pairs). Instead, ISAB introduces a small set of inducing points.

What are inducing points?

They are a fixed number of learnable vectors (I1, I2, I3) that are part of the model's weights, just like the weights in a linear layer. They start random and are updated during pretraining via backpropagation.

Think of them as learned questions that the model asks about any column it sees:

  • I1 might learn to ask "what is the mean of this column?"
  • I2 might learn to ask "how spread out are the values?"
  • I3 might learn to ask "do the y=0 and y=1 values cluster separately?"

We don't hand-design these questions — the model discovers what's useful during training.

The two-step process

  1. Step A — inducing points read the column. Each Ik attends to all 3 column values [0.3, −0.7, 0.1]. It computes attention weights (e.g. I1 might put weight 0.5 on 0.3, 0.3 on −0.7, 0.2 on 0.1) and produces a weighted mixture. After this step, each inducing point holds a specific summary of the column.
  2. Step B — column values read the summaries back. Now each original value attends to the 3 inducing points. The vector for 0.3 looks at I1, I2, I3, decides which summaries are most relevant to it, and mixes them into itself.

After both steps, every vector is enriched with information about the full column. The total cost is only n×m attention pairs (3×3 = 9 here) instead of n² (9) — the savings grow with larger tables.

What changed?

Before ISAB, the vector for 0.3 only encoded "I am the value 0.3 and my row has label y=0." After ISAB, it also encodes "I'm in a column whose values range from −0.7 to 0.3, the y=0 row has the largest value, and the y=1 row has the smallest." This distribution awareness is what makes the final embedding useful.

5

Compute learned scale and bias

The ISAB output is not used directly as the embedding. Instead, two separate projections convert it into per-value scale (W) and bias parameters: 

W    = LayerNorm( Linear( ISAB_output ) )   ← scale
bias = LayerNorm( Linear( ISAB_output ) )   ← shift

These are 64-dim vectors — one W and one bias for each cell in the table. They encode what the ISAB learned about the column distribution and the labels.

6

Final embedding = value × scale + bias

The original scalar value is multiplied element-wise by the learned scale and the learned bias is added. This produces the final 64-dim column embedding for each cell.

0.3 × [ 0.8, 1.1, −0.3, … ]W + [ 0.1, −0.2, 0.5, … ]bias = [ 0.34, 0.13, 0.41, … ]64-dim embedding

Key insight: the embedding preserves the original value but modulates it based on what the ISAB learned about the column distribution and the training labels. Different columns get different W and bias even though they share the same encoder.

Result of Stage 1

Every single cell in the original table is now a 64-dimensional vector. The table went from shape (3 rows, 2 features) to (3 rows, 2 features, 64 dims).

These embeddings flow into Stage 2, where features within each row will interact for the first time.

Stage 2

Features talk to each other.

Until now, x1 and x2 have been processed separately. Stage 2 lets the feature embeddings within each row interact, then compresses each row into a single vector.

0

Input from Stage 1

Each row now has one 64-dim embedding per feature. We'll follow row 1 (train, y=0).

Row 1 x1 emb64 x2 emb64

Each row is processed independently — rows don't see each other yet (that happens in Stage 3).

1

Prepend learnable CLS tokens

4 learnable vectors (CLS1–CLS4) are prepended to the feature embeddings. Like inducing points in Stage 1, these are part of the model's weights — they start random and are trained via backpropagation.

CLS164 CLS264 CLS364 CLS464 x1 emb x2 emb

The row now has 6 tokens: 4 CLS + 2 features, each a 64-dim vector.

Why 4 CLS tokens instead of 1? Multiple CLS tokens let the model capture different aspects of the row (think: one might focus on feature magnitudes, another on feature interactions). Their outputs are concatenated later.

2

Self-attention: every token sees every token

All 6 tokens go through a standard transformer block (self-attention + feed-forward). Every token attends to every other token in the same row.

What this does concretely for our example:

  • The x1 embedding attends to x2. If x1 and x2 are correlated (they are — the synthetic prior generates correlated features), the attention weight will be high, and x1's vector will absorb information about x2.
  • Each CLS token attends to both features, picking up a weighted summary of the row's content.

After this block, x1's embedding no longer represents just "feature 1" — it also knows about feature 2. Cross-feature relationships (e.g. "x1 is high while x2 is low") are now encoded.

3

CLS tokens aggregate the row

A final transformer block runs, but only the 4 CLS tokens are updated. They attend to all 6 tokens (CLS + features) as context, but the feature tokens are not updated.

queries CLS1 CLS2 CLS3 CLS4
keys/values CLS1 CLS2 CLS3 CLS4 x1 x2

This is cross-attention: the CLS tokens are the queries, all 6 tokens are keys and values. Each CLS token produces a final 64-dim summary of the row.

4

Concatenate CLS outputs

The 4 CLS outputs (each 64-dim) are concatenated into a single vector and passed through LayerNorm.

CLS164 ⊕ CLS264 ⊕ CLS364 ⊕ CLS464 → [ … ]256-dim row vector

4 × 64 = 256 dimensions. The feature embeddings are discarded — all information is now compressed into this single row vector.

✓

Result of Stage 2

Every row is now a single 256-dim vector that encodes all its features and their interactions. The table went from (3 rows, 2 features, 64 dims) to (3 rows, 256 dims).

Row 1 [ … ]256 train, y=0
Row 2 [ … ]256 train, y=1
Row 3 [ … ]256 test, y=?

These 3 row vectors flow into Stage 3, where the test row will attend to the training rows to make a prediction.

Stage 3

Learn from context.

This is where in-context learning happens. The test row looks at the labeled training rows — like reading examples before answering a question — and produces a prediction without any gradient update.

0

Input from Stage 2

Three 256-dim row vectors. The model doesn't know which rows are train or test yet — they're just vectors in a sequence.

Row 1 [ … ]256 train
Row 2 [ … ]256 train
Row 3 [ … ]256 test
1

Add label encoding to train rows (again)

Just like in Stage 1, the training labels are one-hot encoded, projected to 256 dims via a new learned linear layer Linear(C, 256), and added element-wise to the train row vectors.

Row 1 [ … ] +
y=0 [ … ]256
= [ … ]
Row 2 [ … ] +
y=1 [ … ]256
= [ … ]
Row 3 [ … ] no label → unchanged

Why add labels a second time? Stage 1 labels were injected per-column before set attention. Here, labels are injected into the full row representation before row-level attention. Both injections use separately learned projections.

2

Attention: test rows can only see train rows

The 3 row vectors go through 3 transformer blocks, but with a crucial constraint: the keys and values are always the train rows only.

This means every row — including the test row — attends exclusively to the 2 labeled training rows. The test row can never attend to itself.

What does this look like concretely?

In each block, the test row (Row 3) computes attention over the 2 training rows:

Row 3 (query) attends to Row 1y=0 Row 2y=1

The attention weights might look like:

weights Row 1: 0.35 Row 2: 0.65

Row 3 is more similar to Row 2, so it assigns higher weight there. The output is a weighted mixture: 0.35 × Row 1 + 0.65 × Row 2 (plus the feed-forward network on top).

This happens 3 times (3 blocks), each refining the representation. After all blocks, the test row's vector is heavily shaped by the training rows it attended to.

This is the in-context learning. The model figures out which training examples are most relevant to the test row and blends their information — including their labels — into the test row's representation. No gradients. No weight updates. Pure attention.

Why is this like an LLM?

In a language model, each new token can only attend to previous tokens (causal mask). Here, the test row can only attend to training rows. The labeled training data is the "prompt," and the test row is the "next token" being predicted from context.

3

MLP decoder: vector → prediction

After the 3 attention blocks, only the test row's vector is kept. It goes through LayerNorm and a two-layer MLP: 

Linear(256, 512) → GELU → Linear(512, C)

For binary classification (C=2, or in practice max_classes=10), the output is a vector of logits — one raw score per class.

Row 3 [ … ]256 → MLP → [ 1.82, −0.41, … ]logits

Softmax converts logits to probabilities:

softmax P(y=0) = 0.10   P(y=1) = 0.90prediction

The model predicts class 1 with 90% confidence. Because Row 3 attended mostly to Row 2 (y=1) and their features were similar, this makes sense.

✓

Result of Stage 3 — and the full model

The test row gets a class prediction. Train rows are discarded — they served purely as context.

Row 3 → predicted class: 1  (confidence 90%)

The complete pipeline: each cell got a column-aware embedding (Stage 1), features interacted within each row (Stage 2), then the test row read the labeled training rows and an MLP produced a prediction (Stage 3). No weights were updated — this was a single forward pass through a pretrained model.

During pretraining, the model sees thousands of synthetic tasks like this and learns to extract patterns from context. At inference, it generalizes to new tables it has never seen.

Run it yourself.

The entire TabICLv2 implementation lives in a single Python file. Pretrain on synthetic tasks and evaluate in minutes. 

uv sync
uv run python -m nanotabicl.demo          # random-weight demo
uv run python -m nanotabicl.nano_model     # pretrain + eval
from nanotabicl import NanoTabICLClassifier, sample_scm_bnn_task

X, y = sample_scm_bnn_task(n_rows=640, n_features=6,
                           task="classification", seed=0)

clf = NanoTabICLClassifier(checkpoint="nano_tabicl.pt")
clf.fit(X[:512], y[:512])
pred = clf.predict(X[512:])
proba = clf.predict_proba(X[512:])
